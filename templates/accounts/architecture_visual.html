🎥 Camera Service Architecture

  • CameraManager: Singleton pattern managing all camera streamers
  • Background Threads: One thread per camera for continuous frame processing
  • Frame Processing: RTSP → OpenCV → Resize (640x360) → JPEG (quality 60)
  • Auto-reconnect: Handles connection failures automatically
  • Inactivity Timeout: Stops after 90 seconds of no requests
  • Thread-safe: Uses locks for frame access

📹 WebRTC Meeting Flow

  • Step 1: Users connect via WebSocket to meeting room
  • Step 2: Exchange SDP offers and answers for capability negotiation
  • Step 3: Exchange ICE candidates for NAT traversal
  • Step 4: Establish P2P connection (direct between users)
  • Step 5: Stream audio/video directly (RTP/UDP)
  • Signaling: All done through Django Channels WebSocket

📦 Protocol & Bandwidth Details

  • HTTP Requests: 1-100 KB per request (pages, API calls)
  • WebSocket Messages: 1-5 KB per message (JSON format)
  • RTSP Stream: 300-600 KB/s per camera (H.264 codec)
  • MJPEG Stream: 300-500 KB/s per mobile camera
  • WebRTC Media: 500 KB/s - 2 MB/s per peer (VP8/VP9, Opus)
  • Database Queries: <10ms response time (local SQLite)

🔐 Permission System

  • Admin: Full access to all cameras and features
  • Teacher: Access only to cameras with explicit permission
  • Student: View all active cameras (no permission check)
  • Database Check: CameraPermission table for teacher access
  • Session-based: Uses Django session authentication
