1. PROJECT OVERVIEW
================================================================================

Project Name: EduMi (Educational Meeting Interface)

Purpose:
EduMi is a comprehensive educational platform designed for schools and 
universities to facilitate real-time video conferencing, camera monitoring, 
and user management. The platform enables teachers to conduct virtual classes, 
monitor campus security cameras, and manage student interactions in a secure 
and scalable environment.

Target Users:
- Teachers: Create and host virtual meetings, manage cameras, view analytics
- Students: Join virtual classes, participate in meetings, view profiles
- Administrators: Manage users, cameras, meetings, and system-wide settings

Key Features:
- Real-time video conferencing with WebRTC technology
- RTSP camera integration for campus monitoring
- Mobile camera support (IP Webcam for Android, DroidCam for iPhone)
- Role-based access control with granular permissions
- Live chat during meetings
- Screen sharing capabilities
- User profile management with avatars
- Meeting scheduling and management
- Multi-camera live monitoring dashboard

3.1 MICROSERVICES DESIGN
--------------------------------------------------------------------------------

The application follows a microservices architecture with two main services:

Service 1: Main Application (Port 8000)
- Purpose: Core application logic, authentication, meetings, UI
- Server: Daphne (ASGI)
- Protocol: HTTP + WebSocket
- Responsibilities:
  * User authentication and authorization
  * Meeting creation and management
  * WebRTC signaling via WebSocket
  * User profile management
  * Camera management UI
  * Dashboard and analytics
  * Template rendering

Service 2: Camera Streaming Service (Port 8001)
- Purpose: Dedicated video streaming from RTSP and mobile cameras
- Server: Django development server (WSGI)
- Protocol: HTTP (MJPEG streaming)
- Responsibilities:
  * RTSP camera connection and streaming
  * Mobile camera (HTTP/MJPEG) streaming
  * Video frame processing with OpenCV
  * Automatic reconnection on failure
  * Frame rate and quality optimization
  * Resource-intensive video operations

Why Separate Services?
1. Isolation: Video streaming is resource-intensive and isolated from main app
2. Scalability: Camera service can be scaled independently
3. Stability: Camera failures don't affect main application
4. Protocol: ASGI (WebSocket) and WSGI don't conflict
5. Performance: Dedicated resources for video processing

Communication Between Services:
- Main app proxies camera feeds from camera service
- HTTP requests from main app (port 8000) to camera service (port 8001)
- CORS enabled for cross-origin requests
- Shared SQLite database for data consistency

3.3 MVT (MODEL-VIEW-TEMPLATE) STRUCTURE
--------------------------------------------------------------------------------

Models (Data Layer):
- Define database schema using Django ORM
- Handle data validation and business logic
- Relationships: OneToOne, ForeignKey, ManyToMany
- Located in: <app>/models.py

Views (Controller Layer):
- Process HTTP requests
- Implement business logic
- Query models and prepare data
- Return HTTP responses or render templates
- Located in: <app>/views.py

Templates (Presentation Layer):
- HTML files with Django template language
- Dynamic content rendering
- Template inheritance (base.html)
- Located in: templates/<app>/

URL Routing:
- Map URLs to views
- App-level: <app>/urls.py
- Project-level: school_project/urls.py

3.5 AUTHENTICATION SYSTEM
--------------------------------------------------------------------------------

Type: Session-based authentication (Django default)

Components:
- Django's built-in User model
- UserProfile model (OneToOne with User)
- Session cookies for authentication state
- CSRF tokens for form security

Authentication Flow:
1. User submits login form (username + password)
2. Django authenticates credentials
3. Session created and cookie sent to browser
4. Subsequent requests include session cookie
5. AuthenticationMiddleware populates request.user
6. Views check user.is_authenticated and permissions

User Roles:
- Admin: Full system access (username='Admin' or is_superuser=True)
- Teacher: Create meetings, manage cameras with permissions
- Student: Join meetings, view cameras

Role Implementation:
- UserProfile.user_type field ('teacher' or 'student')
- Permission checks in views using decorators and conditionals
- Role-based redirects after login

Password Security:
- Django's PBKDF2 algorithm with SHA256 hash
- Automatic password hashing on user creation
- Password validation rules (length, complexity)

Session Security:
- SESSION_COOKIE_SECURE: False (dev), True (production)
- CSRF_COOKIE_SECURE: False (dev), True (production)
- SESSION_COOKIE_SAMESITE: 'Lax'
